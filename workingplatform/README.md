# 动量工作空间

一个本地优先的类 Notion 工作空间原型，用于管理生活、学生工作、学习科研、课外实践等专区中的项目、工作流、时间节点和独立书签库。

## 当前版本

- 使用网页运行，直接打开 `index.html` 即可使用。
- 数据保存在浏览器 `localStorage` 中，适合作为初始本地版本；公开默认项目放在 `data/projects.js`，公开默认日程放在 `data/schedules.js`，本地私有日程可放在不会提交的 `data/schedules.local.js`，默认书签放在 `data/bookmarks.js` 中。
- 支持专区切换、独立书签库、项目列表、工作状态流转、颜色与图标选择、月历节点展示。
- 支持导出 JSON，便于未来迁移到云端数据库。

## 技术路线建议

当前原型为了方便立即使用，采用原生 HTML、CSS、JavaScript 编写，无需安装依赖。进入正式开发后，建议升级为：

- 前端：Next.js + React + TypeScript
- 样式：Tailwind CSS 或 CSS Modules
- 数据库与同步：Supabase
- 认证：Supabase Auth 或 Auth.js
- 部署：Vercel

当前代码已将基础数据拆分为独立数据文件，后续迁移时可以把 `localStorage` 读写替换为 API 或 Supabase 客户端调用，页面渲染逻辑可以继续复用。

这套组合符合当前主流 Web 应用开发方式，迁移成本较低，也适合后续扩展到多人协作、跨设备同步、权限控制、附件管理和全文搜索。

## 云端储存建议

优先推荐 Supabase。

理由：

- 提供 PostgreSQL 数据库，结构化管理专区、项目、任务、书签更稳定。
- 自带用户认证和行级权限，适合未来做个人账号与团队空间。
- 前端调用简单，适合从当前本地 JSON 数据模型平滑迁移。
- 免费层足够支撑原型和早期个人使用。

其他可选方案：

- Firebase：实时同步能力强，但复杂查询和关系型数据不如 PostgreSQL 直观。
- Notion API：适合与 Notion 互通，但不适合作为完整自研应用的主数据库。
- 自建服务器 + PostgreSQL：控制力最强，但维护成本更高。
